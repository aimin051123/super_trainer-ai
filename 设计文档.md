**各层职责**：
- UI 层只做渲染，不写业务逻辑
- 业务层只处理数据流，不碰界面也不直接写 SQL
- 数据层只做存取，不管数据怎么用
- AI 层只负责和 DeepSeek 通信，不管业务上下文

### 1.2 从单文件到模块化的演进

最初所有代码在 `app.py` 一个文件里，约 1200 行。随着功能增加，改一个 bug 要在上千行里翻找。

拆分策略：按照"谁和谁一起变化"来分组——文件解析和知识提取总是一起改，放 `harvest/` 包；出题和薄弱判定总是一起改，放 `quiz/` 包。最终拆成 8 个包，每个包 2-6 个文件，`main.py` 只做 UI 编排。

```
代码/
├── main.py                  ← 所有 UI，360 行
├── config/settings.py       ← 全局常量、API Key 管理
├── database/                ← SQLite 操作（5 文件）
├── ai/                      ← DeepSeek 调用（2 文件）
├── harvest/                 ← 文件处理流水线（6 文件）
├── compile/                 ← 知识编译（2 文件）
├── qa/                      ← 知识问答（2 文件）
├── quiz/                    ← 出题与薄弱（2 文件）
└── plan/planner.py          ← 复习计划
```

## 二、页面架构

### 2.1 四个页面的切换方式

侧边栏顶部放一个下拉框，四个选项对应四个页面。选不同选项时改变 `st.session_state.page` 的值，触发页面刷新，主内容区根据 `page` 值渲染对应内容。

```python
# main.py L76-80
page = st.selectbox("页面", ["📊 总览", "📚 知识库", "🎯 练习", "📅 规划"])
page = page.replace("📊 ","").replace("📚 ","").replace("🎯 ","").replace("📅 ","")
if page != st.session_state.page:
    st.session_state.page = page
    st.rerun()
```

主内容区用四个 `if` 块各自渲染（L125/165/200/318）。页面内容完全独立，改一个页面不影响其他页面。

### 2.2 每个页面的功能

| 页面 | 行号 | 包含的组件 |
|---|---|---|
| 📊 总览 | L125-162 | 四个统计卡片（`st.metric`）+ 掌握度进度条（`st.progress`）+ 薄弱点排序列表 + 今日打卡任务（可点击切换） |
| 📚 知识库 | L165-197 | 文件上传区 + 碎片输入框 + 状态筛选/搜索 + 知识点纯文本列表 + 知识问答（SSE 流式） |
| 🎯 练习 | L200-315 | 学科/模式/题量选择器 + 逐题答题区（radio + 提交 + 进度条）+ 成绩总结 + 错题本（含复习模式） |
| 📅 规划 | L318-355 | 每日时长设置 + 各科考试日期折叠面板 + 自定义要求输入 + 掌握度总览 + 薄弱点列表 + 计划生成按钮 |

### 2.3 CSS 样式

在 `main.py` L17-48 通过 `st.markdown` 注入 4 条样式规则：

- **按钮**：蓝色渐变背景 + 圆角 10px + 阴影 + 鼠标悬停时上浮 + 点击时缩小
- **统计卡片**：白底 + 圆角 14px + 阴影 + 悬停时阴影加深
- **进度条**：圆角 8px
- **背景**：主区域浅灰 `#f5f7fa`，侧边栏白色

选择器使用 Streamlit 的 `data-testid` 属性而非自动生成的 class 名，保证版本升级时样式不会断裂。

## 三、模块实现

### 3.1 数据层（database/）

5 个文件，每个负责一张或一组表的操作：

| 文件 | 负责的表 | 关键操作 |
|---|---|---|
| `connection.py` | 全部 | `db_query()` 执行 SQL，`init_db()` 建 6 张表 |
| `course.py` | course | 增删改查，`course_update()` 更新考试日期和时长 |
| `knowledge_point.py` | knowledge_point | `kp_upsert()` 插入或更新，`kp_list()` 列表查询 |
| `wrong_answer.py` | wrong_answer | `wrong_record()` 记录错题，`wrong_review()` 复习判定 |
| `chat_history.py` | chat_history | `chat_save()` 保存对话，`chat_recent()` 取最近 10 条 |

`db_query()` 每次调用新建一个 SQLite 连接，执行完自动关闭。对于需要"插入后立即获取 ID"的场景（如 `kp_upsert`），使用单连接确保 `last_insert_rowid()` 取到正确的值。

### 3.2 AI 层（ai/）

`ai/client.py` 提供统一入口 `call_ai(prompt, system_prompt, json_mode, stream, temperature)`。内部流程：读取 API Key → 创建 httpx 客户端（配置代理）→ 调用 DeepSeek API → 返回结果或错误信息。

`ai/stream.py` 提供 `stream_display()`，用于问答场景的逐字显示。每收到 3 个 token 或遇到中文标点时刷新一次界面，避免频繁重渲染。

### 3.3 收割模块（harvest/）

6 个文件组成一条流水线，处理文件从上传到入库的全过程：

```
parse_file()           →  提取文字（PDF/PPTX/DOCX/TXT）
classify_subject()     →  判断学科（先调 AI，失败用关键词词典兜底）
slice_text()           →  按标题正则切片（Markdown标题 / 数字序号 / 中文章节）
save_raw()             →  逐片存为 知识库/原始资料/{学科}/时间戳_标题.md
extract_knowledge_points() →  AI 读前 4000 字提取知识点列表，写入 knowledge_point 表
```

`classify_subject()` 的回退机制：AI 调用返回 JSON → 解析 `subject` 字段 → 成功返回。失败则遍历预置的关键词词典（8 个学科各 7-10 个关键词），数命中次数，选最多的。全不匹配返回"其他学科"。

### 3.4 编译模块（compile/）

`compile/compiler.py` 负责把原始碎片整理成结构化知识页面。流程：

1. 扫描 `知识库/原始资料/` 下所有 .md 文件
2. 读已编译页面的 `source` 字段，排除已处理过的文件
3. 对每个新文件调 AI，prompt 中包含已有页面列表和去重规则
4. AI 返回 JSON：`{title, subject, page_type, is_new, merge_with, content, linked_pages}`
5. `is_new=false` 且 `merge_with` 非空 → 合并到已有页面（追加内容，更新 updated 时间）
6. `is_new=true` → 创建新页面，写入 `知识库/编译后知识/{学科}/页面名.md`
7. 全部编译完后扫描 `linked_pages`，在被引用页面末尾追加"## 被以下页面引用"列表

### 3.5 问答模块（qa/）

`qa/retriever.py` 负责检索。遍历 `知识库/编译后知识/` 下所有 .md 文件，对每个文件统计用户问题中每个字的出现次数。标题命中额外 +10 分。取得分最高的前 5 个页面。

`qa/answer.py` 负责回答。流程：检索 → 取最近 10 条对话历史 → 拼 system prompt（角色 + 参考资料 + 历史对话）→ 调 `call_ai(stream=True)` → `stream_display()` 逐字渲染 → 保存对话到 `chat_history` 表。

### 3.6 出题模块（quiz/）

`quiz/generator.py` 的 `generate_quiz()` 支持四种模式：

| 模式 | 实现方式 |
|---|---|
| 薄弱点优先 | 查 `is_weak=1` 的知识点名称，拼接 `"优先针对以下薄弱知识点出题：xxx, yyy"` 注入 prompt |
| 全面练习 | 不注入薄弱点信息，知识页内容随机出题 |
| 专项突破 | 接收 `focus_kp_ids` 参数，只保留用户选中的知识点 |
| 模拟考试 | 传 `subject=None`，遍历所有有资料的学科综合出题 |

出题 prompt 包含完整的知识点列表（ID + 名称 + 掌握度% + 薄弱标记），要求 AI 每道题返回 `knowledge_point_id` 字段。返回后验证 ID 是否真实存在（防 AI 幻觉），不存在则按 `source_page` 字符串匹配或自动创建。

`quiz/weakness.py` 的 `judge_weak()` 实现 4 条规则的短路判断。每条规则检查一个条件，命中即返回，不继续检查后续规则。`kp_update_mastery()` 在每次答题后调用，重算掌握度并更新 `knowledge_point` 表。

### 3.7 计划模块（plan/）

`plan/planner.py` 的 `generate_plan()` 接收四个参数：课程 ID、每日时长、自定义要求（选填）、各科考试日期字典。内部收集各科日期和剩余天数、知识点掌握度分布、薄弱点列表，拼成 prompt。prompt 中包含艾宾浩斯复习间隔规则。AI 返回 Markdown 格式计划文本。

## 四、数据库

### 4.1 六张表

```
course ──< knowledge_point    （课程下有哪些知识点）
course ──< quiz_record        （课程下有哪些测验记录）
course ──< quiz_question      （课程下每道题的明细）
```

### 4.2 关键字段速查

| 表 | 关键字段（索引） | 用途 |
|---|---|---|
| knowledge_point | `[5]=mastery` `[8]=is_weak` `[9]=weak_reason` | 掌握度、薄弱标记 |

| wrong_answer | `review_count` `mastered` | 复习次数、是否已掌握 |

## 五、数据流

### 5.1 用户上传一份 PDF 发生了什么

```
用户拖入文件
  → main.py L167-172 接收文件对象
  → harvest_file() 调用 parse_file() 提取全部文字
  → classify_subject() 判断学科（AI → 关键词 → "其他学科"）
  → 如果是新学科，自动调用 course_create() 创建对应课程
  → slice_text() 按标题正则切成 N 个片段
  → save_raw() 逐片存为 .md 文件
  → extract_knowledge_points() 调 AI 提取知识点 → kp_upsert() 写入数据库
  → 页面显示 "✅ 已收割→【学科】N个片段，提取了M个知识点"
```

### 5.2 用户刷一组题发生了什么

```
选择学科/模式/题量 → 点击生成
  → generate_quiz() 调 AI 出题 → 验证 ID → 存 session_state
  ↓
逐题展示：显示题目 → 用户选答案 → 点提交
  → 本地比对 ul[0]==cl[0] → 显示对错和解析
  ↓
全部完成 → 点"完成测验"
  → 创建 quiz_record（测验快照）
  → 逐题写入 quiz_question（关联知识点 ID）
  → kp_update_mastery() 更新掌握度 + 判薄弱
  → 答错的题 → wrong_record() 写入错题表
```

### 5.3 用户生成复习计划发生了什么

```
规划页设置各科日期和时长 → 点生成
  → generate_plan() 收集数据 → 拼 prompt → 调 AI
  → AI 返回 Markdown 计划文本
  → 自动提取薄弱点前 4 个作为今日打卡任务
  → 总览页读取打卡任务，用户可逐项勾选完成
```

## 六、容错处理

### 6.1 AI 调用失败时的降级

每个 AI 调用点都有备选方案：

| 功能 | 主方案 | 备选方案 |
|---|---|---|
| 学科分类 | AI 返回 JSON | 关键词词典匹配 |
| 碎片类型 | AI 返回 JSON | 硬编码关键词（区别→对比，步骤→算法） |
| 知识点提取 | AI 返回 JSON | 返回 0，跳过这一步 |
| 知识编译 | AI 返回 JSON | 跳过该文件，继续下一个 |
| 出题 | AI 返回 JSON | 显示"解析失败，请重试" |
| 问答 | AI 流式返回 | 显示错误文本 |

### 6.2 防 AI 幻觉

AI 出题时可能返回不存在的知识点 ID。提交答案前验证 ID 是否在数据库中存在：
1. 不在 → 置为 None
2. 按 `source_page` 字段匹配已有知识点名称
3. 匹配不上 → 调用 `kp_upsert()` 自动创建新知识点

### 6.3 已修复的关键问题

| 问题 | 原因 | 修复 |
|---|---|---|
| 数据库 ID 取不到 | `db_query()` 每次新建连接，`last_insert_rowid()` 跨连接返回 0 | 需要 ID 的场景改用单连接执行 |
| 知识点索引写错 | `r[4]` 是 importance 字段，`r[5]` 才是 mastery | 统一使用正确的索引位置 |
| 编译失败但显示成功 | `except: continue` 吞掉错误 | 改为显式记录失败原因 |
| 页面空白 | CSS 复杂选择器冲突 | 精简为 4 条规则 |

| 场景 | temperature | json_mode | 说明 |

| 学科分类 | 0.3 | 开启 | 需要精确分类，低温度 |

## 八、模块间调用关系

```
main.py
  ├── harvest/ → ai/ + database/
  ├── compile/ → ai/ + harvest/
  ├── qa/      → ai/ + database/ + compile/
  ├── quiz/    → ai/ + database/ + compile/ + harvest/
  └── plan/    → ai/ + database/ + compile/
```

所有模块最终都依赖 `ai/` 和 `database/` 这两个底层包。业务模块之间不互相依赖，避免循环 import。
