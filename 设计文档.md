## 第零章：功能目标速览（给时间有限的人）

### 0.1 一句话定义

SuperTutor 是一个基于 AI 的期末复习工具。你给它课件，它拆成知识点；你刷它出的题，它告诉你哪里弱；你告诉它哪天考试，它排一份每天干什么的计划。

### 0.2 项目背景

大学生期末复习和考研备考普遍面临四个痛点：
- 学习资料分散（教材、笔记、课件），缺乏统一管理和结构化整理
- 难以定位自身薄弱知识点，复习缺乏针对性
- 缺乏动态调整的学习计划，容易陷入盲目刷题或无效重复
- 传统问答机器人只能被动回答，无法基于用户自己的资料给出准确答案

### 0.3 七个功能各解决什么问题

| 功能 | 解决什么问题 | 达到什么效果 |
|---|---|---|
| **知识收割** | 资料散落在 PDF/PPT/Word 里，从未被统一管理 | 拖进去 → AI 按学科分类 → 按章节切片 → 存入对应文件夹 |
| **碎片收割** | 灵感稍纵即逝，记在备忘录里永远不被翻出来复习 | 粘贴一段笔记 → AI 识别学科和类型 → 和正式资料存在一起 |
| **知识编译** | 碎片格式混乱（口语笔记 + 干教材 + PPT 要点），AI 消化不了 | AI 重写成格式统一、互相链接的结构化页面 |
| **知识问答** | 直接问 ChatGPT 跟教材说法不一致，还可能编造 | 先检索你的知识库 → AI 基于你的资料回答 → 标注来源 |
| **精准刷题** | 题库和教材脱节，难度无法调节，不知道自己哪里弱 | AI 基于你的资料出题 → 支持薄弱点优先 → 自动批改 |
| **薄弱点判断** | "感觉自己不会"不可靠，没有数据支撑 | 4 条规则自动判定 → 告诉你"为什么薄弱"而不只是"薄弱" |
| **复习规划** | 手动排计划要么太松完不成，要么太紧撑不住 | AI 根据考试日期+掌握度+遗忘曲线生成每日计划 → 刷题后自动调整 |

### 0.4 学习闭环

```
上传资料 → AI提取知识点 → 编译为知识库
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
           知识问答        精准刷题         复习计划
           (基于资料)     (基于知识点)     (基于掌握度)
                              │               │
                              ▼               │
                    薄弱点判断 ←─────────────┘
                              │
                    下次出题优先出薄弱点 ──→ 循环
```

### 1.2 每个模块"做什么"与"为什么不能被替换"

#### UI 层：Streamlit 网页前端

**作用**：把 Python 函数的结果变成浏览器里的按钮、输入框、标签页、进度条，让用户不写一行 HTML/CSS 就能操作整个系统。


---

#### 知识收割模块

**包含函数**：`parse_file`、`slice_text`、`classify_subject`、`harvest_file`、`harvest_fragment`、`extract_knowledge_points`

**作用**：所有"把外部资料变成系统可处理格式"的工作——解析 PDF/PPT/Word → 切片 → 判断学科 → AI 提取结构化知识点 → 存入文件系统和数据库。


---

#### 知识点提取模块

**包含函数**：`extract_knowledge_points`

**作用**：上传资料后，AI 从全文提取结构化的知识点列表（名称 + 描述 + 重要度），存入 `knowledge_point` 表。这是整个系统从"文档层"上升到"认知层"的桥梁。



---

#### 知识编译模块

**包含函数**：`compile_knowledge`、`update_backlinks`、`update_index`、`append_changelog`

**作用**：把原始碎片（口语化笔记 + 教材摘录 + PPT 要点）AI 重写为格式统一、层次分明、互相链接的知识页面，并建立双向链接网络。


---

#### 精准刷题模块

**包含函数**：`generate_quiz`、`judge_weak`、`update_mastery`

**作用**：整个系统的"诊断引擎"——基于知识点出题 → 本地批改（不调 AI）→ 更新掌握度 → 写入答题记录。


---

#### 错题管理模块

**包含函数**：`record_wrong`、`review_wrong`、`mark_mastered`

**作用**：让错题从"答题流水里勾出来看一眼"升级为"有独立生命周期的复习对象"——支持重新作答、隐藏正确答案、标记已掌握、追踪复习次数。


---

#### 复习规划模块

**包含函数**：`generate_plan`、`adjust_plan`、`detect_weak_points`

**作用**：把考试日期、剩余天数、每日时长、各知识点掌握度、薄弱点列表拼成 prompt，让 AI 按艾宾浩斯遗忘曲线排出一份每日计划。测验后自动刷新。


---

#### AI 调用层：call_ai()

**作用**：统一封装所有对 DeepSeek API 的调用——配置读取（三级回退）、超时处理、错误捕获、JSON 模式、SSE 流式。

## 第二部分：数据库设计

### 2.1 功能-数据映射

| 功能 | 涉及的表 | 表的职责 |
|---|---|---|
| 课程管理 | `course` | 存课程名、考试日期、每日时长 |
| 上传资料 / 知识点提取 | `knowledge_point` | 存 AI 提取的知识点、掌握度、薄弱标记 |
| 出题 | `knowledge_point` + `quiz_question` | 出题时查知识点列表做薄弱优先，题目存 `quiz_question` 并外键关联知识点 |
| 批改 | `quiz_question` + `knowledge_point` | 更新每道题的用户答案，重算知识点掌握度 |
| 测验历史 | `quiz_record` | 存一次测验的整体信息（题数、正确数、时间） |
| 错题复习 | `wrong_answer` | 独立管理错题生命周期（复习次数、是否已掌握） |
| 知识问答 | `chat_history` | 存对话记录，下次提问时取最近 5 轮作为上下文 |

一句话概括：**`course` 是一切的外键锚点，`knowledge_point` 是掌握度的核心载体，`quiz_question` 存题目并与知识点绑定，`wrong_answer` 让错题成为可复习对象，`chat_history` 支撑多轮对话记忆。**

### 2.2 ER 关系

```
course (1) ────< (N) knowledge_point
course (1) ────< (N) quiz_record
course (1) ────< (N) quiz_question
course (1) ────< (N) wrong_answer
course (1) ────< (N) chat_history
quiz_record (1) ────< (N) quiz_question
knowledge_point (1) ────< (N) quiz_question
knowledge_point (1) ────< (N) wrong_answer
```

### 2.3 每张表的设计理由

| 表 | 为什么需要 | 关键字段 |
|---|---|---|
| `course` | 所有数据按课程隔离。同一台电脑上考两门课，答题记录互不干扰 | `exam_date`、`daily_study_minutes` |
| `knowledge_point` | AI 提取的结构化知识点。掌握度、薄弱标记都挂在这张表上 | `mastery`、`is_weak`、`weak_reason` |
| `quiz_record` | 一次测验的整体快照。查"最近一次测验的正确率"一行 SQL 搞定 | `quiz_type`、`total_questions`、`correct_count` |
| `quiz_question` | 每道题的完整记录。`knowledge_point_id` 外键把题目和知识点精确绑定 | `knowledge_point_id`（核心外键） |
| `wrong_answer` | 错题的独立生命周期。支持复习次数追踪、掌握标记、从列表消失 | `review_count`、`mastered` |
| `chat_history` | 多轮对话的上下文记忆。每次提问取最近 10 条一起发给 AI | `role`、`content` |

### 2.4 关键设计原则

**为什么用外键而不是字符串匹配**：`quiz_question.knowledge_point_id` 是整数外键，不是页面名字符串。页面改名不影响历史数据，一个知识点可以被多道题关联，掌握度按 ID 精确汇总。

**为什么错题独立成表**：错题有独立生命周期（复习次数、掌握标记），和答题流水是不同的业务对象。混在一起的结果是"复习一道错题做对了，但它仍躺在错误列表里"。

**为什么测验拆成 record + question 两张表**：一张表既存整体信息又存每题细节，查"最近一次测验的正确率"需要手动分组。拆表后各司其职。

## 第四部分：图片类资料识别方案

### 4.1 问题

系统当前仅支持电子版资料（PDF/PPTX/DOCX/TXT/MD）。以下资料无法处理：
- 手机拍的板书照片（JPG/PNG）
- 扫描版 PDF（每页本质是一张图片）
- 教材截图

### 4.2 推荐方案：清晰度检测 + 多模态 AI

```
图片类文件
  ├── 清晰度检测通过 → 多模态 AI 直接识别
  └── 清晰度不通过 → 拒绝，提示用户重新拍摄清晰照片
```

**核心思路**：清晰的照片值得用 AI 识别（保证准确率），模糊的照片直接拒绝（比识别出一个错误结果更好）。

**多模态 AI 选型**：当前 DeepSeek 无公开多模态 API，需接入 GPT-4V 或开源模型（Qwen-VL）。

### 4.3 技术约束

- 图片识别的准确率不可能达到 100%，系统需保留原始图片文件供人工核对
- 一张清晰板书照片的识别成本约 ¥0.01-0.03（多模态模型 token 消耗较高）

**API 端点**：`POST https://api.deepseek.com/v1/chat/completions`  
**认证**：`Authorization: Bearer <DEEPSEEK_API_KEY>`  
**配置读取**：`config.properties` → 环境变量 `DEEPSEEK_API_KEY` → 侧边栏输入（三级回退）  
**模型**：`deepseek-chat`

### 5.2 SSE 流式优化

Streamlit 的 `st.markdown()` 每次调用都会完整序列化 + WebSocket 传输 + DOM 替换，没有真正的 diff 优化。每收到一个 token 就刷新会导致约 800 次重渲染。

优化方案——批量刷新：

```python
buffer = []
for token in stream:
    buffer.append(token)
    full_text += token
    # 每 3 个 token 或遇到标点才刷新 UI（减少 70% 重渲染）
    if len(buffer) >= 3 or token in "，。！？\n":
        placeholder.markdown(full_text + "▌")
        buffer = []
```

## 第六部分：关键技术决策

| 决策 | 选了 | 没选 | 为什么 |
|---|---|---|---|
| 网页框架 | Streamlit | Flask / Django | 不用写 HTML/CSS/JS，纯 Python |
| 数据库 | SQLite | MySQL / PostgreSQL | 不用安装服务，拷贝文件夹就能迁移 |
| 知识存储 | 文件夹 + .md | 全部存数据库 | 用户可手动打开 .md 看，零门槛 |
| AI 调用库 | openai 官方库 | requests 手写 | 自带重试、超时、SSE 解析 |
| 出题类型 | 只有选择题 | 判断/填空/简答 | 选择题批改最简单可靠，其他题型需语义判断 |
| 检索方式 | 关键词匹配 | 向量数据库 | 几百个页面不需要向量，避免部署复杂度 |
| 代码风格 | 纯函数 | 面向对象 | 函数 = 输入→输出，新手不需要理解继承和多态 |
| 文件组织 | 单文件 app.py | 多文件拆分 | 新手在一个文件里修 bug 比在 5 个文件间跳来跳去轻松 |
| 薄弱点判定 | 4 条规则 if-else | AI 判断 | 规则简单，AI 不可控（两次调用可能不同结论） |
| 批改方式 | 本地字符串比对 | 再调 AI 批改 | 出题时 AI 已返回答案，再调一次是浪费 |

## 第七部分：错误处理策略

**核心原则**：任何一步出错，提示用户原因和解决方案，绝不出现 Python 红色报错堆栈。

```
调用 AI
  ├── API Key 没配 → "请创建 config.properties 文件并填入密钥"
  ├── 网络超时（30秒）→ "调用超时，请检查网络后重试"
  ├── AI 返回的不是 JSON → 回退到关键词匹配（学科分类）/ 提示重试（出题）
  └── 未知异常 → "AI 调用失败：{异常信息}"，程序继续运行

读数据库
  ├── 表不存在 → init_db() 在启动时自动建表
  └── 查询结果为空 → 返回空列表，UI 显示"暂无数据"

文件解析
  ├── PDF 文本提取为空 → "该 PDF 可能为扫描件，无法提取文字"
  ├── 不支持的文件格式 → "仅支持 PDF / PPTX / DOCX / TXT / MD"
  └── 文件编码不匹配 → UTF-8 失败自动尝试 GBK

知识库为空
  └── 刷题和问答功能显示"请先上传资料并编译知识"
```
