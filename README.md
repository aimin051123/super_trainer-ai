## 1. 项目概述

### 1.1 项目定位

基于多智能体协作的智能复习系统。学生上传PDF教材后，系统自动完成知识解析与向量索引、智能出题、自动批改、错因诊断和SM-2间隔重复排期。

### 1.2 核心技术指标

| 指标 | 目标值 |
|------|--------|
| 文档解析速度 | 不超过30秒/50页 |
| 问答响应时间 | 不超过3秒 |
| 批改准确率 | 不低于90% |
| 考点提取准确率 | 不低于85% |
| 向量检索精度 | Top-5召回率不低于80% |
| 数据存储 | 完全本地化，无云端上传 |

### 1.3 技术范围

**包含**: PDF解析引擎、语义切片与向量化、向量检索系统、多智能体协作框架、状态机编排、SM-2排期算法、错因诊断分类器

**不包含**: 移动端应用、多租户系统、知识图谱可视化、OCR识别、实时协作

## 4. 多智能体协作设计

### 4.1 Agent定义

| Agent | 角色 | 职责 | 输入 | 输出 | 算力档位 |
|-------|------|------|------|------|----------|
| Tutor | 主导师 | PDF解析、知识切片、向量索引、SM-2排期 | PDF文件路径 | 切片列表 + 向量索引 + 复习计划 | medium |
| Assistant | 助教 | 基于知识库生成测验题目 | 知识点列表 + 难度参数 | JSON格式试题集 | medium |
| Evaluator | 评估者 | 批改作答、迷思概念诊断、苏格拉底式引导 | 学生答案 + 标准答案 | 批改结果 + 诊断标签 + 引导提示 | heavy |

### 4.2 Agent协作流程

| 阶段 | 触发动作 | 执行Agent | 核心操作 | 输出 |
|------|----------|-----------|----------|------|
| PARSING | start() | Tutor | PyMuPDF文本提取、语义切片、Embedding向量化、sqlite-vec索引写入 | 切片列表 + 向量索引 |
| QUIZ_GEN | proceed() | Assistant | 向量库检索知识点、LLM生成题目（布鲁姆分层） | 10道选择题 |
| EVALUATING | submit_answers() | Evaluator | 选择题即时判定、简答题LLM批改、错因诊断分类（7类标签）、苏格拉底式提示生成（3层） | 批改结果 + 诊断标签 + 引导提示 |
| PLANNING | proceed() | Tutor | EMA平滑更新掌握度、SM-2间隔重复算法计算复习日期 | 复习计划 |

### 4.3 状态机设计

| 状态 | 说明 | 可执行动作 | 下一状态 |
|------|------|------------|----------|
| IDLE | 空闲，等待上传 | start() | PARSING |
| PARSING | 解析中 | get_status() | PARSED |
| PARSED | 解析完成 | proceed() | QUIZ_GEN |
| QUIZ_GEN | 出题中 | get_questions() | QUIZ_READY |
| QUIZ_READY | 等待作答 | submit_answers() | EVALUATING |
| EVALUATING | 批改中 | get_results() | EVALUATED |
| EVALUATED | 批改完成 | proceed() | PLANNING |
| PLANNING | 排期中 | get_plan() | DONE |
| DONE | 完成 | - | - |

状态转换由Orchestrator统一管理，每个状态转换触发对应的Agent执行。

### 4.4 Agent间通信机制

采用共享状态池模式。所有Agent通过Orchestrator访问统一的State对象:

```
{
  material_id: "uuid",
  chunks: [...],
  knowledge_points: [...],
  questions: [...],
  answers: [...],
  mastery_records: {...},
  study_plan: {...}
}
```

Agent间不直接通信，通过读写共享状态实现协作，降低耦合度。


## 5. 数据模型

| 实体 | 主要字段 | 说明 |
|------|----------|------|
| Material | id, file_name, file_size, status, created_at | 上传的PDF材料 |
| KnowledgeChunk | id, material_id, content, embedding, metadata | 知识切片，含向量 |
| KnowledgePoint | id, name, difficulty, frequency, description | 提取的知识点 |
| Question | id, kp_id, type, content, options, answer, explanation | 题目，含解析 |
| QuizSession | id, material_id, questions_json, status, created_at | 测验会话 |
| AnswerRecord | id, session_id, question_id, user_answer, is_correct, answer_time | 作答记录 |
| WrongQuestion | id, user_id, question_id, wrong_answer, diagnosis_label, socratic_hints_json | 错题与诊断 |
| MasteryRecord | id, user_id, kp_id, accuracy, history_json, last_tested | 掌握度追踪 |
| StudyPlan | id, user_id, kp_id, scheduled_date, status, interval_days | 复习计划任务 |

### 5.2 向量存储设计

使用sqlite-vec扩展实现本地向量检索:

```sql
-- 创建虚拟表
CREATE VIRTUAL TABLE chunks_vec USING vec0(
  id INTEGER PRIMARY KEY,
  embedding FLOAT[768]
);

-- 向量检索 (余弦相似度)
SELECT id, distance 
FROM chunks_vec 
WHERE embedding MATCH ? 
ORDER BY distance 
LIMIT 5;
```

索引策略:

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 向量维度 | 768 | BAAI/bge-large-zh-v1.5 |
| 相似度度量 | 余弦距离 | sqlite-vec默认 |
| 检索方式 | 多路召回 | 向量 + 关键词BM25 |
| 重排序 | 交叉编码器精排 | 可选，提升精度 |

## 10. 技术难点与创新点

### 10.1 核心技术难点

| 难点 | 挑战 | 解决方案 |
|------|------|----------|
| 状态机编排 | 多Agent协作需要精确控制状态流转 | LangGraph实现声明式状态图，支持断点续传 |
| LLM输出解析 | 生成题目和诊断需要结构化输出 | Pydantic模型约束 + JSON模式强制 + 重试机制 |
| 向量检索精度 | 语义相似度匹配直接影响出题质量 | 多路召回(向量+BM25) + 交叉编码器重排序 |
| 掌握度建模 | 客观量化学生知识水平 | EMA指数平滑 + SM-2间隔重复算法 |
| 错因分类 | 7类标签的准确识别依赖推理能力 | 精心设计的System Prompt + few-shot示例 |

### 10.2 技术创新点

| 创新点 | 技术说明 |
|--------|----------|
| 认知孪生建模 | 为每个知识点维护掌握度曲线，基于作答历史动态更新 |
| 三档算力调度 | 根据任务复杂度动态选择LLM模型(heavy/medium/light)，优化成本 |
| 苏格拉底式诊断 | 不直接给答案，通过3层渐进式提示引导学生发现错误 |
| 嵌入式向量检索 | sqlite-vec实现向量检索零运维，无需单独部署向量数据库 |
| 间隔重复排期 | 基于遗忘曲线的SM-2算法，自动计算最优复习间隔 |
